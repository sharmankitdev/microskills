# DAG-RULES ŌĆö hand-authoring a `WORKFLOW.yaml`

> The complete, authoritative reference for the constructs that make up a workflow DAG in this
> repo. If you want to write a `WORKFLOW.yaml` by hand (instead of generating one with
> `/workflow-create`), this is your spec. Every rule here is enforced by `validate-workflow`,
> `compile-workflow`, or the `workflow` dispatcher ŌĆö the source of truth, not aspiration.

**Schema:** [`templates/references/workflow-schema.json`](./templates/references/workflow-schema.json) ┬Ę
**Compiler:** [`catalog/scripts/compile-workflow`](./catalog/scripts/compile-workflow) ┬Ę
**Validator:** [`catalog/scripts/validate-workflow`](./catalog/scripts/validate-workflow) ┬Ę
**Skeleton:** [`templates/workflow-template.yaml`](./templates/workflow-template.yaml) ┬Ę
**Runtime:** `catalog/skills/workflow/SKILL.md`

## 1. Mental model

A workflow is a **declarative DAG**. You describe *what* runs and *what depends on what*; the
compiler decides *how* it runs.

```
WORKFLOW.yaml  ŌöĆŌöĆcompile-workflowŌöĆŌöĆŌ¢Č  .compiled/seg-N.js   (autonomous background segments)
                                       .compiled/manifest.json  (the run plan)
                                            Ōöé
                                            Ō¢╝
                                   workflow dispatcher (skill)
                                   runs segments on the native Workflow engine,
                                   pauses at checkpoints (gates + orchestrator nodes)
```

Two execution worlds, and the compiler decides which world each node lives in:

| World | What runs there | Can it pause for a human? |
|---|---|---|
| **Background segment** | `use:` microskills + `agent:` nodes, fanned-out and looped, on the native Workflow engine | **No.** No `AskUserQuestion`. A subagent that tries to ask will silently fabricate. |
| **Orchestrator checkpoint** | human-approval gates + orchestrator-native nodes, in the main loop | **Yes.** Filesystem side-effects, `AskUserQuestion`, nested-workflow calls live here. |

The compiler **partitions the topologically-sorted DAG into maximal background segments separated
by orchestrator checkpoints.** A node never crosses that boundary ŌĆö interaction *only* happens at
checkpoints, between segments.

**Determinism is the whole point.** Same `WORKFLOW.yaml` + same profile ŌåÆ **byte-identical**
compiled output, every time. (Topological ties break by definition order; stale segment files are
deleted before each compile.) You get reliable orchestration from the *compile step*, not from
hoping an LLM behaves.

## 2. File layout

A workflow definition is a directory under `harness/workflow-defs/<name>/` (committed source) which
`harness-sync` materializes into `.claude/workflow-defs/<name>/` (the runtime the compiler reads):

```
<name>/
  WORKFLOW.yaml            # the DAG (this document is about this file)
  profiles/
    base.yaml              # OPTIONAL overlay merged onto WORKFLOW.yaml; names the default profile (absent ŌåÆ empty overlay)
    <overlay>.yaml         # optional named profiles (e.g. autonomous.yaml)
  references/              # optional prose contracts the nodes point at
  .compiled/               # generated by compile-workflow (gitignored): seg-*.js + manifest.json
```

Validate and compile from the project root:

```bash
.claude/scripts/validate-workflow .claude/workflow-defs/<name>/WORKFLOW.yaml \
    .claude/workflow-defs/<name>/profiles/base.yaml          # 0 = pass, 1 = block
.claude/scripts/compile-workflow <name> [--profile <p>]       # writes .compiled/
```

## 3. Top-level keys

The grammar is **closed** ŌĆö `additionalProperties: false` everywhere. An unknown key fails schema
validation. Only `name` + `nodes` are required; everything else is optional.

| Key | Required | Purpose |
|---|---|---|
| `name` | Ō£ģ | Bare identifier, `^[a-z][a-z0-9-]*$` (kebab-case). Names the def dir and the compiled segments (`<name>__segN`). |
| `nodes` | Ō£ģ | The DAG vertices. Ōēź1. Each is a `use:`, an `agent:`, a `workflow:`, or an orchestrator node. |
| `version` | ŌĆö | **Now optional.** Defaults to `1` (compile/validate `setdefault` it before schema validation). An *explicit* value other than `1` still blocks. |
| `description` | ŌĆö | One-line summary. Surfaced in the manifest + dispatcher recap. |
| `vars` | ŌĆö | Intra-def variable map (see below). `{{key}}` substitution; profile-overridable; no file include. |
| `inputs` | ŌĆö | Caller-supplied typed inputs. Referenced as `${workflow.inputs.<name>}`. |
| `gates` | ŌĆö | Checkpoints anchored `after:` a node. Hard / human-approval gates become segment boundaries. |
| `loop` | ŌĆö | At most one. A guarded loop (`while` *or* `until`) over a **contiguous** subset of nodes. |
| `output` | ŌĆö | `{from: <node-id>}` ŌĆö whose result is the workflow's aggregate output. |
| `profile` | ŌĆö | `base.yaml` only: `{default: <name>}` names the default overlay when `--profile` is omitted. |
| `imports` | ŌĆö | **Now load-bearing.** The allowlist of child workflows a `workflow:` node may invoke (see below). |
| `base` | ŌĆö | **Advisory only.** Catalog-curation hint for `initialize-harness`; ignored by the compiler. |

### `vars` ŌĆö intra-def substitution

```yaml
vars:
  model: opus
  cap: 3
nodes:
  - id: plan
    agent: planner
    prompt: Plan with {{model}}.        # {{model}} ŌåÆ opus before schema validation
loop:
  max_iters: 3                          # (a {{cap}} token anywhere would resolve to 3)
```

A `{{key}}` token **anywhere** in the merged doc (values *and* dict keys) is textually replaced from
the `vars:` map by **both** `compile-workflow` and `validate-workflow`, **after** deep-merge +
strip-nulls (so a profile overlay may override a var) and **before** schema validation. Values coerce
to string at substitution time. It is a **pure textual transform** ŌĆö *not* `$include`, no cross-file
mechanism. An unresolved `{{key}}` (no matching var) is **left intact** and surfaced as a `warn`
(stderr from compile; a `warn` issue from validate) ŌĆö never a block. A doc with no `{{...}}` tokens
compiles byte-identically.

### `imports` ŌĆö the child-workflow allowlist

```yaml
imports: [build-workflow-from-plan]     # every `workflow:` node's target MUST be listed here
```

`imports` is the **load-bearing allowlist** of child workflows a `workflow:` node (┬¦5d) may invoke:
each bare child name a `workflow:` node targets **must** appear here, or validate (with `--defs-root`)
blocks it. It also drives validate's nested-target resolution and import-cycle detection. A legacy
`<dir>/<name>` advisory form is still *tolerated* by the schema (un-migrated defs) but is **ignored**
by the nested-resolution pass. It still has no effect on the compiled segment bytes.

### `inputs`

```yaml
inputs:
  requirement:                 # referenced as ${workflow.inputs.requirement}
    type: string
    required: true
    description: Natural-language description of what to build.
  staging_dir:
    type: string
    required: false
    default: .claude/.workflow-staging/current   # used when caller omits it
    description: Sandbox dir for staged files.
```

`required: true` inputs land in the manifest's `required_inputs` (the dispatcher must collect them).
Inputs with a `default` land in `input_defaults`. `type`/`description` are documentation ŌĆö not
type-checked at runtime.

## 4. The reference grammar ŌĆö `${...}`

Anywhere a string appears (prompts, `inputs` values, `when`, `for_each`, loop `while`/`carry`) you
can embed references. There are these forms:

| Form | Means | Compiles to (inside a segment) |
|---|---|---|
| `${workflow.inputs.<x>}` | A workflow-level input | `_args.wf_<x>` |
| `${<node-id>.output.<field>...}` | An upstream node's output field | `n_<id>.<field>` if same segment, else `_args["<id>"].<field>` |
| `${<node-id>.items}` | The **whole per-item-results array** of a `for_each` node | the producer's bare array handle: `n_<id>` if same segment, else `_args["<id>"]` |
| `${loop.carry.<v>}` | A loop carry variable (loop body only) | `carry_<v>` |
| `${<as>}` | The current `for_each` item (the `as` binding) | the bare loop variable |

**`${<id>.items}`** consumes the *entire* fan-out result array of `for_each` node `<id>` ŌĆö there is
**no `.field` tail**, the node's result *is* the array. It implies a dependency edge into the
referencing node exactly like `${<id>.output}` (┬¦7). The producer `<id>` **must** be a `for_each`
node ŌĆö a `.items` ref to a non-fan-out node is a block, an unknown id is a block. (The compiler
resolves `.items` *before* `.output` so a producer field literally named `items` still compiles.)

**Gate ids are legal ref targets.** `${<gate-id>.output.choice}` resolves against the human's recorded
pick at a gate (┬¦8). A gate-id ref is *accepted* by validate (not "unknown node") and ŌĆö unlike a node
ref ŌĆö creates **no** dependency edge (a gate isn't a node), so it can never perturb ordering.

A reference may be a **whole value** (`requirement: ${workflow.inputs.requirement}`) or **embedded
in text** (`prompt: > process ${ms_spec.name} at ${plan.output.path}`). In prompts and other text,
each ref is wrapped in a `__ref()` helper that JSON-stringifies non-strings, so an object ref
interpolates cleanly.

### Refs imply edges (you no longer restate them in `depends_on`)

> **A `${<id>.output...}` or `${<id>.items}` reference now IMPLIES a dependency edge `<id> ŌåÆ this`.**

The effective dependency set of a node is the **union** of its explicit `depends_on` and every
`.output`/`.items` ref in its scoped fields (`inputs`, `prompt`, `when`, `for_each`). You do **not**
have to restate a ref in `depends_on` anymore ŌĆö the union is what the compiler topo-sorts and what
both tools cycle-check (┬¦7). `depends_on` remains useful for *pure ordering* edges that carry no ref
(e.g. `build` waits for `provision`).

The validator still guards genuine typos: a ref to a node that doesn't exist is a **block**
(`references output of unknown node '<id>'`). A `${workflow.inputs.x}` ref is a workflow input, not a
dependency on a node named `workflow`; `${<gate-id>.output.choice}` is the accepted gate-id form and
implies no edge.

## 5. Nodes ŌĆö the four kinds

Every node has a unique `id` (`^[a-z][a-z0-9_-]*$`). A node is **exactly one of four kinds** (`use`,
`agent`, `workflow` are mutually exclusive ŌĆö declaring more than one is a block; the fourth kind
declares none of them):

### (a) `use:` ŌĆö invoke a registered microskill

```yaml
- id: plan
  use: task-plan                       # bare microskill name; resolved via resolve-microskill
  customize: { profile: microskill }   # selects which profile of the microskill to run
  inputs:
    requirement: ${workflow.inputs.requirement}
    staging_dir: ${workflow.inputs.staging_dir}
  output_schema:                       # the contract later nodes reference
    type: object
    required: [plan_path, scope_advisory]
    properties:
      plan_path:      { type: ["string", "null"] }
      scope_advisory: { type: ["object", "null"] }
```

The compiler resolves the microskill's directives and **bakes its executor identity** (the
`runtime.agent` / `runtime.model` from the resolved profile) into the call, so the node runs *as*
that agent on that model. (Background segments cannot nest-dispatch, so the executor must *be* that
agent.)

### (b) `agent:` ŌĆö invoke an agent directly with a prompt

```yaml
- id: analyze
  agent: workflow-planner              # an agent type
  inputs:
    monolith_path: ${workflow.inputs.path}
  prompt: >
    Analyze the monolith at ${workflow.inputs.path} and produce a decomposition brief.
  output_schema:
    type: object
    required: [decomposition_requirement]
    properties:
      decomposition_requirement: { type: string }
```

`agent:` nodes are **always background** (autonomous) unless a gate forces a checkpoint.

### (c) Orchestrator-native ŌĆö neither `use:` nor `agent:`

```yaml
- id: finalize
  delegation: orchestrator             # required: a node with no use/agent is only valid as orchestrator
  depends_on: [plan, implement, evaluate]
  prompt: >
    Vendor the approved artifact into harness/, upsert the manifest, run harness-sync,
    and AskUserQuestion if there's a name conflict.
```

Orchestrator nodes run in the **main loop**: they do filesystem side-effects, `AskUserQuestion`, and
**nested-workflow invocation**. They become checkpoints in the compiled sequence.

### (d) `workflow:` ŌĆö invoke a nested workflow (first-class)

```yaml
imports: [build-workflow-from-plan]    # the child MUST be allowlisted here
nodes:
  - id: build
    workflow: build-workflow-from-plan  # bare child-workflow name (kebab-case)
    depends_on: [plan]
    inputs:
      plan_path: ${plan.output.plan_path}   # raw refs ŌĆö resolved by the dispatcher at run time
```

A `workflow:` node is its own kind ŌĆö a **first-class nested-workflow call**. It is **always an
orchestrator checkpoint** (`checkpoint_type: nested_workflow`); the compiler classifies it as
orchestrator *before* any use/agent inspection, because a background segment can't re-enter the
dispatcher. It carries `inputs` / `depends_on` / `when` / `for_each` + `as` like any node; its
`inputs` values stay as raw `${...}` refs in the manifest and the dispatcher resolves them against
workflow inputs + upstream results when it re-enters the `workflow` skill for the child.

**Profile passthrough.** A `customize: { profile: <name> }` on the node selects **the child
workflow's** profile: the compiler stamps `profile: <name>` into the checkpoint and the dispatcher
compiles the child with `--profile <name>` (absent ŌåÆ child compiles with its default, and the
checkpoint stays byte-identical). This is how `provision` runs `microskill-create` with the
`autonomous` profile so the child's plan gate never pauses for a human.

With `validate-workflow --defs-root <dir>`, the child contract is checked against `imports` **and**
the child's `WORKFLOW.yaml`: the target must be in `imports` (else block), must resolve to a child
`WORKFLOW.yaml` (else block), every `required` child input must be supplied by this node's `inputs`
(else block), a `customize.profile` (when set) must resolve to a child `profiles/<name>.yaml` (else
block), and `${<this-node>.output.<field>}` refs are cross-checked against the child's declared
output node (mismatch ŌåÆ **warn**). **Nesting depth is 1**: a child that itself contains a `workflow:`
node is blocked, and an import cycle across the reachable import graph is blocked. Without
`--defs-root` these checks are skipped (backward-compatible single-file validation).

### Full node field reference

| Field | Applies to | Meaning |
|---|---|---|
| `id` | all | Unique, `^[a-z][a-z0-9_-]*$`. Must be disjoint from gate ids. |
| `use` | (a) | Bare microskill name. Mutually exclusive with `agent`/`workflow`. |
| `agent` | (b) | Agent type. Mutually exclusive with `use`/`workflow`. |
| `workflow` | (d) | Bare child-workflow name (kebab-case). Mutually exclusive with `use`/`agent`; must be in `imports`. Always an orchestrator checkpoint. |
| `delegation` | all | `orchestrator` forces a checkpoint; `auto` forces background. Overrides the auto-classification. |
| `side_effect` | all | `true` is a **readable alias for `delegation: orchestrator`** ŌĆö normalized to it before classification (an explicit `delegation` already present wins). Use it to mark a node that does filesystem side effects / human interaction. |
| `depends_on` | all | Upstream node ids ŌåÆ DAG edges. Unioned with ref-implied edges (┬¦7). |
| `inputs` | (a)(b)(d) | Map of input field ŌåÆ `${...}` expression (or literal). Wires data in. |
| `prompt` | (b)(c) | Task prompt template. May contain `${...}`. (For `use:` nodes the body comes from the microskill.) |
| `output_schema` | (a)(b) | JSON-Schema fragment forcing structured output. Passed to `agent()`'s `schema`. A `use:` node **omitting** this inherits the microskill's resolved schema (┬¦11). |
| `when` | all | Conditional guard (see ┬¦6). |
| `for_each` + `as` | all | Fan-out over a collection (see ┬¦6). |
| `max_parallel` | for_each only | Bound the fan-out to batches of K (`parallelChunked`) instead of one unbounded `parallel()`. **Only valid on a `for_each` node** ŌĆö on a non-fan-out node it is a block. |
| `customize` | (a)(d) | On a `use:` node, `{profile: <name>}` selects the microskill's profile; on a `workflow:` node, it selects the **child workflow's** profile (passed through to the nested-workflow checkpoint). **See the footgun in ┬¦13.** |
| `grant_tools` | (a)(b) | **Declared but inert** ŌĆö the compiler does not consume it. See ┬¦13. |

## 6. Control flow inside the DAG

### `when` ŌĆö conditional skip

A node runs only if `when` (an inner-of-`${...}` JS expression) is truthy; otherwise it yields
`null`.

```yaml
- id: implement
  use: task-implement
  when: ${plan.output.scope_advisory == null}    # main path
  depends_on: [plan]

- id: advise
  delegation: orchestrator
  when: ${plan.output.scope_advisory != null}    # advisory path ŌĆö the mutually-exclusive fork
  depends_on: [plan]
  prompt: > Present the planner's advisory and stop.
```

Inside a background segment, `when` compiles to a ternary: `(cond) ? await runAgent(...) : null`.
A `${id.output}` ref inside `when` implies the dependency edge (the edge is real even though the node
may skip ŌĆö ┬¦7), so you needn't restate it in `depends_on`. For **orchestrator** nodes the dispatcher
evaluates `when` in the main loop: false ŌåÆ store `null`, continue.

> **Advisory-fork pattern** (used by every built-in creator): two nodes guarded by opposite
> conditions (`== null` vs `!= null`) so exactly one runs. The main path builds; the advisory path
> surfaces guidance and stops.

### `for_each` + `as` ŌĆö fan-out

Run one instance per item in a collection. `as` names the per-item loop variable, referenced as
`${<as>}`.

```yaml
- id: notify-owners
  delegation: orchestrator
  for_each: ${plan.output.changed_files}   # a collection
  as: file                                 # binds each item
  depends_on: [plan]
  prompt: >
    Notify the owner of ${file.path} that it changed.
```

- `for_each` **requires** `as`, and `as` must be a safe identifier `^[a-z][a-z0-9_]*$` (it becomes a
  JS variable). Missing `as` or a bad identifier is a **block**. `as` without `for_each` is a
  **warn** (no effect).
- In a background segment, `for_each` compiles to `await parallel((collection || []).map((<as>) => () => ...))`
  ŌĆö **fan-out runs in parallel**. An empty/absent collection ŌåÆ empty result.
- **`max_parallel: K`** (for_each only) caps the fan-out to fixed-size batches of K via a
  `parallelChunked` helper (emitted into the segment only when a `max_parallel` node is present), each
  batch awaited in turn ŌĆö output order is preserved, identical to the unbounded `parallel()`. Without
  it, the emission is byte-identical to before. `max_parallel` on a node with no `for_each` is a
  **block** in both validate and compile.
- The whole fan-out result array is consumable downstream via **`${<id>.items}`** (┬¦4).
- **`for_each` is not allowed inside a `loop.body`** ŌĆö fan-out inside a loop is a **block**
  (`fan-out inside a loop body is unsupported`).

## 7. Edges, ordering, cycles

- The DAG edge set is the **UNION** of explicit `depends_on` **and** every `${<id>.output...}` /
  `${<id>.items}` ref in a node's scoped fields (`inputs`/`prompt`/`when`/`for_each`). A ref therefore
  **implies an edge** ŌĆö you no longer have to restate it in `depends_on` (┬¦4). (Only refs whose id is
  a real node count; a gate-id ref contributes no edge.)
- The compiler runs **Kahn's topological sort over the union edge set**; ties break by **definition
  order** (this is what makes output deterministic). Author your nodes in a sensible order and the
  compiled sequence follows it. Compile and validate share the *same* `union_edges`/`topo_sort`, so
  they can never disagree on ordering.
- A **dependency cycle is a block** (both validator and compiler reject it):
  `dependency cycle: a -> b -> a`. Cycle detection runs over the **union** edges, so a *ref-only*
  cycle (A reads `${B.output}`, B reads `${A.output}`, neither in `depends_on`) is still caught.
- A `depends_on` entry that names a non-existent node is a **block** (`references unknown node`).
- **Typed field-ref check:** an undeclared `${<producer>.output.<field>}` where `<producer>` has an
  `output_schema` of `type: object` with declared `properties` and `<field>` is **not** among them is
  a **block**. A schema-less producer (or one without an object schema) is treated as `any` and never
  flagged.
- Orchestrator nodes may have `depends_on` too ŌĆö even though they all run in the main loop, the
  edges enforce ordering (e.g. `build` waits for `provision` so provisioned components exist first).

## 8. Gates ŌĆö checkpoints

A gate is a checkpoint anchored **after** a node.

```yaml
gates:
  - id: approve_plan
    after: plan                  # fires after this node; must be a known node id
    type: human_approval         # human_approval | verification | tool_check
    severity: hard               # hard (default) | warn
    prompt: Review the plan. Approve to implement, revise to re-plan, or abandon.
    options: [approve, revise, abandon]   # choices for a human_approval gate
```

**What makes a gate split a segment:** a gate of `type: human_approval` **or** `severity: hard`
(hard is the default) flushes the current background segment and becomes an orchestrator checkpoint.
A `severity: warn` gate does **not** pause ŌĆö the dispatcher renders the output, emits the prompt, and
continues.

| Field | Rule |
|---|---|
| `id` | `^[a-z][a-z0-9_-]*$`. **Must be unique among gates AND disjoint from node ids** (they share the `${id...}` ref namespace) ŌĆö a collision is a block. |
| `after` | Required. Must reference a known node (`'after' references unknown node` otherwise). |
| `type` | `human_approval` \| `verification` \| `tool_check`. `human_approval` **requires** a `prompt`. |
| `severity` | `hard` (default) \| `warn`. Hard ŌåÆ segment boundary + pause. |
| `prompt` | The question shown to the human. |
| `options` | Choices for `human_approval` (e.g. `[approve, revise, abandon]`). The dispatcher maps these to consequences: approve ŌåÆ continue; revise ŌåÆ re-run the segment with notes; abandon ŌåÆ stop + clean staging. |

> Only `human_approval` gates are exercised by the built-in workflows. `verification` / `tool_check`
> are valid `type` values in the schema; a `hard`-severity gate of any type still forms a segment
> boundary.

### Gate-choice branching

The dispatcher records the human's pick at a gate as `results[<gate-id>] = { choice: <selected
option> }` (the chosen label, verbatim ŌĆö recorded for `warn` gates too). A downstream node can branch
on that pick:

```yaml
gates:
  - id: review
    after: plan
    type: human_approval
    prompt: Approve, or send to deep review?
    options: [approve, deep_review]
nodes:
  - id: deep_pass
    agent: reviewer
    when: ${review.output.choice == 'deep_review'}    # branch on the human's pick
    depends_on: [plan]
    prompt: Do the deep review of ${plan.output.spec}.
```

A `${<gate-id>.output.choice}` ref is **accepted** by validate (treated as `any`, never field-checked)
and creates **no dependency edge** (a gate isn't a node), so it can't perturb ordering. This composes
with the advisory-fork pattern ŌĆö opposite-`choice` guards keep the branches mutually exclusive.

## 9. `loop` ŌĆö guarded iteration

At most one `loop` per workflow. It repeats a contiguous subset of nodes until a condition flips or a
cap is hit.

```yaml
loop:
  while: ${!evaluate.output.pass}    # continue-condition over the latest iteration's output
  # or, equivalently:
  # until: ${evaluate.output.pass}   # STOP-condition sugar ŌĆö desugars to while: ${!(evaluate.output.pass)}
  max_iters: 3                       # MANDATORY cap
  body: [implement, evaluate]        # node ids that repeat each iteration (must be contiguous)
  carry:
    last_findings: ${evaluate.output}  # state threaded into the next iteration
```

- The schema requires **`max_iters` + `body`** (no longer `while`). It's an integer cap Ōēź 1
  (`missing required 'max_iters' cap` otherwise).
- **Exactly one of `while` / `until`** must be present (enforced by compile + validate, not the
  schema): neither ŌåÆ block, both ŌåÆ block. `until: ${EXPR}` is sugar that **desugars to
  `while: ${!(EXPR)}`** (the inner expression is negated and re-wrapped) ŌĆö the compiled output is
  byte-identical to writing that negated `while` by hand. The desugar runs before every ref /
  contiguity check, so both forms are validated identically.
- `body` lists the node ids that repeat. Each must be a known node; a `for_each` node in the body is
  a **block**.
- The body compiles to a JS `do { ... } while ((<while>) && __iter < max)` inside **one segment**.
- `carry` threads state across iterations: each var inits from `_args.carry_<v>` (`null` on the first
  round) and is reassigned at the end of each iteration. Body nodes read it via `${loop.carry.<v>}`.
- The loop returns a `__rounds` count (the iteration count) so the dispatcher can recap "loop done in
  N round(s)". `__rounds` is **not** a node result.

**Guarded-loop break:** if any body node has a `when`, the loop tracks a `__ran` flag and emits
`if (!__ran) break` ŌĆö an iteration where *every* node skipped exits the loop immediately (instead of
spinning to `max_iters` and reading a skipped node's `null` in the `while`). You don't write this; the
compiler emits it when it sees a `when` in the body.

> ### ŌÜĀ’ĖÅ The loop-contiguity rule (now ENFORCED, formerly a silent footgun)
> **An orchestrator node (or a hard/`human_approval` gate) placed *between* two loop-body nodes splits
> the loop segment and would break the do/while scaffold.** The loop body must compile into a single
> uninterrupted segment. This is **now a block in both `validate-workflow` and `compile-workflow`**
> (shared `loop_contiguity_error`, run over the same topological order): a body that isn't a
> contiguous window ŌĆö an interleaved out-of-body / orchestrator node, or a hard gate anchored after a
> non-last body node ŌĆö fails before any emit. (Previously it compiled to a silently-broken loop.)
> **Keep `loop.body` nodes contiguous in the topological order**, and place orchestrator nodes / gates
> *before* or *after* the loop body ŌĆö never inside it.

## 10. `output`

```yaml
output:
  from: finalize     # the node whose output is the workflow's aggregate result
```

`from` must reference a known node (block otherwise). At the end of the run the dispatcher reports
`results[<from>]` as the workflow's result. If a workflow is invoked by another workflow's
orchestrator node, this is what the caller receives.

## 11. Profiles & customization ŌĆö tune without forking

A `WORKFLOW.yaml` is layered before it compiles:

```
WORKFLOW.yaml  ŌŚĆmergeŌöĆ  profiles/base.yaml  ŌŚĆmergeŌöĆ  profiles/<overlay>.yaml  ŌŚĆapplyŌöĆ  --override a.b=c
```

**Resolution order:** explicit `--profile <p>` flag ŌåÆ `base.profile.default` ŌåÆ `base`. The chosen
overlay (if not `base`) is deep-merged after `base.yaml`. (`base.yaml`'s own `version` is not merged
in.) Only `base.yaml` may carry a `profile:` block ŌĆö an overlay declaring `profile` is a block.
`profiles/base.yaml` is **optional**: an absent file resolves to an empty overlay (the workflow compiles
from `WORKFLOW.yaml` alone), matching the microskill resolver and the optional-`version` ergonomics.

**Deep-merge rules** (shared with microskills, from `resolve-microskill`):

| Case | Rule |
|---|---|
| Key in overlay is `null` | **Delete** the key from base. |
| Both values are maps | **Recurse** (deep-merge). |
| Both values are lists, key path is `gates.add` | **Append** (overlay items concatenated to base). |
| Both values are lists, any other key | **Replace** wholesale. |
| Key path is `output_schema` | **Replace** wholesale (a contract, never a partial merge). |
| Scalars / new keys | **Replace** / add. |

**`--override a.b.c=value`** sets a dotted path; the value is parsed as YAML (so `=true` ŌåÆ boolean,
`=[a,b]` ŌåÆ list, `=` ŌåÆ delete). After all merges/overrides, remaining `null`s are stripped.

**List-verb overlays ŌĆö edit `nodes:` / `gates:` by id.** Beyond wholesale list replacement, a profile
overlay can surgically `add` / `patch` / `remove` workflow `nodes:` and `gates:` by id (the
`resolve-microskill` deep-merge verbs, shared here):

```yaml
# profiles/autonomous.yaml
gates:
  patch: [{ id: approve_plan, severity: warn }]   # soften an existing gate by id
nodes:
  patch: [{ id: implement, customize: { profile: fast } }]
  add:   [{ id: extra_check, agent: checker, depends_on: [implement] }]
  remove: [advise]                                 # drop a node by id
```

`patch` deep-merges into the matching-id item; `add` appends new items; `remove` drops by id. A
`remove` that **orphans a reference** ŌĆö leaving a dangling `depends_on`, ref, `gate.after`,
`output.from`, or `loop.body` entry ŌĆö is caught at validate (as the corresponding unknown-node block).
*(Microskills have the analogous `steps.add/patch/remove` on their step list ŌĆö the microskill-side
equivalent of this same mechanism.)*

**`use:` node output_schema inheritance.** A `use:` node that **omits** `output_schema` now
**inherits** its microskill's fully-resolved (profile-merged, null-stripped) `output_schema` ŌĆö the
compiler stashes it on the node and bakes it into the call. An explicit node-level `output_schema`
wins. Inheritance is a default-for-*absent* only: a microskill profile that *deleted* its schema
(`output_schema: null`) resolves to none, so a deleted schema is never resurrected.

**Node-level `customize`:** on a `use:` node, `customize: { profile: <name> }` selects *which profile
of that microskill* the node runs; on a `workflow:` node, it selects *which profile of the child
workflow* the dispatcher compiles (┬¦5d). `profile` is the **only** `customize` key the compiler reads
(see ┬¦13).

Example ŌĆö the `autonomous` profile of `microskill-create` softens the approval gate from `hard` to
`warn`: the structure stays (2 segments, the gate checkpoint still exists) but the run never pauses
for a human.

## 12. How it compiles & runs (so you can predict the output)

### Classification ŌĆö background vs orchestrator

The compiler classifies each node (`side_effect: true` is normalized to `delegation: orchestrator`
*before* this runs):

1. `delegation: orchestrator` ŌåÆ **orchestrator** (explicit wins). `delegation: auto` ŌåÆ **background**.
2. `workflow:` ŌåÆ **orchestrator** (a nested-workflow checkpoint) ŌĆö classified before any use/agent
   inspection.
3. Neither `use` nor `agent` (nor `workflow`) ŌåÆ **orchestrator** (an orchestrator-native step).
4. `agent:` ŌåÆ **background**.
5. `use:` ŌåÆ resolve the microskill's directives and inspect:
   - resolution **fails** ŌåÆ orchestrator (fail-safe);
   - resolved `allowed_tools` contains **`AskUserQuestion`** ŌåÆ orchestrator;
   - resolved directives carry a **hard-severity gate** ŌåÆ orchestrator;
   - otherwise ŌåÆ **background**.

Under `--explain` the compiler surfaces a per-node **classification reason** (the short string behind
each decision, e.g. `use: task-plan -> background (autonomous)`, `nested workflow: build-ŌĆ”`) so you
can see *why* a node landed where it did.

### Segmentation

Walk the topo order. Background nodes accumulate into the current segment. An orchestrator node, a
`workflow:` node, or a hard/human-approval gate **flushes** the segment and emits a checkpoint
(`orchestrator_node` / `nested_workflow` / `gate` respectively). A loop's body stays inside one
segment (`is_loop` segment). Result: an alternating sequence of `segment[...]` and checkpoint steps,
written to `manifest.json`.

Example compiled sequences (from the real workflows):

```
microskill-create:        segment[plan]  gate  orchestrator_node(advise)
                          segment[implement,evaluate]  orchestrator_node(finalize)
build-workflow-from-plan: segment[implement,evaluate]  orchestrator_node(finalize)
workflow-create:          segment[plan]  gate  orchestrator_node ├Ś3 (provision, advise, build)
```

### Runtime threading (what the dispatcher does)

For each manifest step in order:

- **`segment`** ŌåÆ build `args`: `wf_<x>` for each needed workflow input, `<id>` for each needed
  upstream node output, `carry_<v> = null` for loop carries. Invoke the Workflow engine with the
  segment's `scriptPath` + `args`. The segment returns an object keyed by its `produces` node ids;
  store each into `results`. (A loop segment also returns `__rounds`.)
- **`checkpoint` / gate** ŌåÆ render `results[gate.after]` as readable markdown (never raw JSON), then
  `AskUserQuestion` with the gate's prompt + options. Approve ŌåÆ continue; revise ŌåÆ re-run the
  producing segment with folded-in notes; abandon ŌåÆ stop + clean staging. `warn` gates render +
  continue without pausing.
- **`checkpoint` / orchestrator_node** ŌåÆ execute the node's `prompt` in the main loop, resolving
  `${...}` against `inputs`/`results`. Honor `when` (skip ŌåÆ `null`) and `for_each` (run per item,
  bind `${as}`). This is where side-effects and `AskUserQuestion` happen. Store into `results[node]`.
- **`checkpoint` / nested_workflow** (a `workflow:` node) ŌåÆ honor `when`/`for_each` as above, resolve
  the step's `inputs` refs against `inputs`/`results`, then **re-enter the same `workflow` skill** for
  the child (compile + run its manifest, feeding the resolved inputs and skipping interactive input
  gathering). Depth Ōēż 1 is compiler-guaranteed, so recursion is bounded. Store the child's
  `output.from` result into `results[node]`.

At the end, report `results[manifest.output.from]`.

### `--plan` and `--explain` (compile flags)

- **`compile-workflow ŌĆ” --plan`** ŌĆö a **dry-run**: compute the full plan (segment JS, `needs`,
  `produces`, manifest, summary) but **write nothing** ŌĆö no `mkdir`, no stale-clean, no `seg-*.js`, no
  `manifest.json`. The printed summary is identical to a real compile (plus `plan: true`). Mirrors the
  harness reconcile loops' `--plan`-by-default ergonomics. (The dispatcher never passes `--plan`.)
- **`compile-workflow ŌĆ” --explain`** ŌĆö **output-only**, never changes emitted bytes. Adds: per-node
  `classification` (id ŌåÆ class + reason), a **de-anonymized** `sequence` (checkpoints labelled
  `gate:<id>` / `orchestrator_node:<id>` / `nested_workflow:<id>` instead of the bare lossy type), and
  the **`manifest_hash`** ŌĆö both in the summary *and* written into `manifest.json` (the only thing that
  changes the written bytes vs a default compile). `manifest_hash` is a SHA-256 over the canonical
  (sorted-key, compact) manifest *excluding its own hash field*; any change to the partition / needs /
  produces changes it.

### Dispatcher resume (`.run-state.json`)

After each step the dispatcher checkpoints its `results{}` to
`.compiled/.run-state.json` as `{manifest_hash, step_index, results}`. This is **dispatcher runtime
state, not a compiled artifact** ŌĆö the compiler never reads it, and its stale-clean globs only
`seg-*.js`, so it is never deleted (determinism untouched). On the next run the dispatcher compiles
with `--explain`, then if `.run-state.json` exists **and its `manifest_hash` matches** the freshly
compiled manifest, it offers to **resume** from the stored `step_index` (re-using completed work ŌĆö e.g.
an expensive opus planner segment). A **mismatched** hash (the workflow was recompiled / changed, so
stored node outputs no longer line up) or an absent file ŌåÆ start fresh at step 0.

## 13. Footguns & gotchas

- **ŌÜĀ’ĖÅ Loop contiguity (┬¦9) is now ENFORCED, not silent.** An orchestrator node or hard gate between
  two `loop.body` nodes used to compile to a silently-broken loop; it is now a **block** in both
  validate and compile. Keep body nodes contiguous.
- **ŌÜĀ’ĖÅ Gate / node id collisions are blocked.** Gate ids share the `${id...}` ref namespace with node
  ids, so a gate id must be unique among gates **and** disjoint from every node id ŌĆö a collision is a
  validate block.
- **ŌÜĀ’ĖÅ `customize` only reads `.profile`.** The schema describes `customize` as "profile and/or
  dotted overrides," but the compiler reads **only** `customize.profile`. Arbitrary dotted overrides
  under `customize` are **silently ignored**. To override a microskill's resolved config per-node,
  there is no wired mechanism today ŌĆö author a dedicated profile instead. *(To tune the workflow
  itself, e.g. `loop.max_iters`, use a profile overlay on the workflow's `base.yaml`/named profile,
  applied via `--profile` ŌĆö that path is real.)*
- **ŌÜĀ’ĖÅ `grant_tools` is inert.** It's in the schema but the compiler never consumes it. Don't rely on
  it to grant tools ŌĆö bake tool permissions into the microskill/agent profile.
- **No `AskUserQuestion` in a background segment.** A `use:`/`agent:` node that needs to ask a human
  must be reclassified to orchestrator (it auto-classifies that way if its microskill's
  `allowed_tools` includes `AskUserQuestion` or it carries a hard gate). Asking inside a segment =
  silent fabrication.
- **Nested workflows are checkpoints, never background.** Use the first-class `workflow:` node (┬¦5d)
  ŌĆö it always classifies as an orchestrator checkpoint. Background subagents have no orchestration
  context and can't re-enter the dispatcher. (You can still drive a nested workflow from a plain
  `delegation: orchestrator` node's `prompt`, but `workflow:` gets you the import-allowlist + child-
  contract checks.)
- **`workflow:` targets must be in `imports`.** A `workflow:` node whose target isn't allowlisted in
  `imports` is a validate block (with `--defs-root`); nesting depth > 1 and import cycles are blocked.
- **`output_schema` replaces, never merges.** A profile that overrides `output_schema` must restate
  the *whole* schema ŌĆö base fields do not leak in. (A `use:` node that *omits* its own schema inherits
  the microskill's ŌĆö ┬¦11.)
- **`base` does nothing at runtime;** `imports` is now load-bearing (┬¦3) ŌĆö it's the `workflow:`
  allowlist, not mere documentation.
- **Loop body can't fan out.** `for_each` inside `loop.body` is a hard block.

### YAML-syntax traps (these bite before any DAG rule does)

- **`${...}` cannot live inside a `{ }` flow mapping.** `inputs: { x: ${a.output.y} }` fails to parse ŌĆö
  the `{`/`}` inside `${...}` collide with the flow-map braces. Always write `inputs:` (and any field
  holding a ref) in **block style**:
  ```yaml
  inputs:
    x: ${a.output.y}      # Ō£ģ
  # inputs: { x: ${a.output.y} }   # ŌØī YAML parse error
  ```
  Flow maps are fine when there's no ref inside them (`customize: { profile: microskill }`,
  `output: { from: build }`).
- **`prompt: >` needs its text on the *next* line.** A folded/literal block scalar can't carry inline
  text. Use a plain scalar for one-liners, or break to the next indented line for `>`:
  ```yaml
  prompt: Plan it.              # Ō£ģ plain scalar
  prompt: >                     # Ō£ģ folded ŌĆö text starts next line
    Plan it, using ${plan.output.spec}.
  # prompt: > Plan it.          # ŌØī YAML parse error
  ```
  A ref mid-scalar is fine (`prompt: Build ${plan.output.spec}.`); a value that *starts* with `{` or
  `[` would be read as flow ŌĆö refs start with `$`, so that's a non-issue in practice.

## 14. Worked skeletons

All below are **validated + compiled** ŌĆö each is followed by the `sequence` the compiler emits.

**Linear ŌĆö one segment, no checkpoints ŌåÆ `segment[a,b]`:**
```yaml
version: 1
name: linear-flow
nodes:
  - id: a
    agent: worker
    prompt: Do the first thing.
  - id: b
    agent: worker
    depends_on: [a]
    prompt: Use ${a.output.result} to do the second thing.
output: { from: b }
```

**Gated two-segment ŌåÆ `segment[plan]` ┬Ę gate ┬Ę `segment[build]`:**
```yaml
version: 1
name: gated-flow
inputs:
  requirement: { type: string, required: true }
nodes:
  - id: plan
    agent: planner
    inputs:
      requirement: ${workflow.inputs.requirement}   # block style ŌĆö ${...} can't sit in a { } flow map
    prompt: Plan it.
    output_schema:
      type: object
      required: [spec]
      properties:
        spec: { type: object }
  - id: build
    agent: builder
    depends_on: [plan]
    prompt: Build ${plan.output.spec}.
gates:
  - id: approve
    after: plan
    type: human_approval
    severity: hard
    prompt: Approve the plan?
    options: [approve, abandon]
output: { from: build }
```

**Retry loop with advisory fork ŌĆö the built-in `microskill-create` shape ŌåÆ
`segment[plan]` ┬Ę gate ┬Ę orchestrator(advise) ┬Ę `segment[implement,evaluate]` ┬Ę orchestrator(finalize):**
```yaml
version: 1
name: build-thing
inputs:
  requirement: { type: string, required: true }
nodes:
  - id: plan
    agent: planner
    inputs:
      requirement: ${workflow.inputs.requirement}
    output_schema:
      type: object
      required: [scope_advisory]
      properties:
        scope_advisory: { type: ["object", "null"] }
  - id: advise                                    # advisory fork
    delegation: orchestrator
    when: ${plan.output.scope_advisory != null}
    depends_on: [plan]
    prompt: Surface the advisory and stop.
  - id: implement
    agent: builder
    when: ${plan.output.scope_advisory == null}   # main path
    depends_on: [plan]
    inputs:
      last_findings: ${loop.carry.last_findings}
  - id: evaluate
    agent: checker
    when: ${plan.output.scope_advisory == null}
    depends_on: [implement, plan]
    output_schema:
      type: object
      required: [pass, issues]
      properties:
        pass: { type: boolean }
        issues: { type: array, items: { type: object } }
  - id: finalize                                  # orchestrator: AFTER the loop body, not inside
    delegation: orchestrator
    when: ${plan.output.scope_advisory == null}
    depends_on: [plan, implement, evaluate]
    prompt: Vendor and sync the approved artifact.
gates:
  - id: approve_plan
    after: plan
    type: human_approval
    severity: hard
    prompt: Approve the plan, revise, or abandon.
    options: [approve, revise, abandon]
loop:
  while: ${!evaluate.output.pass}
  max_iters: 3
  body: [implement, evaluate]      # contiguous; finalize is AFTER them
  carry:
    last_findings: ${evaluate.output}
output: { from: finalize }
```

> The `agent:` nodes above keep the skeleton self-contained. To compose registered microskills
> instead, swap `agent: planner` ŌåÆ `use: task-plan` + `customize: { profile: microskill }` (and the
> matching `task-implement` / `task-evaluate`), exactly as the real `microskill-create` does.

**Nested workflow + fan-out ŌĆö plan, then build each item via a child workflow ŌåÆ
`segment[plan]` ┬Ę gate ┬Ę nested_workflow(build):**
```yaml
name: orchestrate                  # version omitted ŌåÆ defaults to 1
imports: [build-workflow-from-plan]   # the child MUST be allowlisted
inputs:
  requirement: { type: string, required: true }
nodes:
  - id: plan
    agent: planner
    inputs:
      requirement: ${workflow.inputs.requirement}
    output_schema:
      type: object
      required: [units]
      properties:
        units: { type: array, items: { type: object } }
  - id: build                       # first-class nested-workflow node ŌåÆ always a checkpoint
    workflow: build-workflow-from-plan
    for_each: ${plan.output.units}  # one child run per unit
    as: unit
    depends_on: [plan]              # ref-implied too; restated here for clarity
    inputs:
      plan_path: ${unit}            # raw refs ŌĆö the dispatcher resolves them at run time
gates:
  - id: approve_plan
    after: plan
    type: human_approval
    prompt: Approve the plan, or revise?
    options: [approve, revise]
output: { from: build }
```
> `build` carries `for_each`/`as` like any node; with `validate-workflow --defs-root <dir>` the child's
> required inputs are checked against this node's `inputs`, and depth>1 / import cycles are blocked.

## 15. Authoring checklist

Before you compile, confirm:

- [ ] `name` is kebab-case, `nodes` is non-empty. (`version` is optional ŌĆö defaults to `1`; an
      explicit non-`1` blocks.)
- [ ] Every node is exactly one of: `use:`, `agent:`, `workflow:`, or orchestrator-native
      (`delegation: orchestrator` / `side_effect: true`) ŌĆö never more than one of use/agent/workflow.
- [ ] Every `${id.output...}` / `${id.items}` ref names a real node (refs imply edges ŌĆö `depends_on`
      is the union, you needn't restate them); typed field refs match the producer's `output_schema`.
- [ ] No dependency cycles (checked over the union edges); no `depends_on` / `gate.after` / `loop.body`
      / `output.from` pointing at a non-existent node.
- [ ] Every `for_each` has a safe-identifier `as`; `max_parallel` only on `for_each` nodes; no
      `for_each` inside `loop.body`; `${id.items}` only targets a `for_each` node.
- [ ] `loop` has `max_iters` + `body` and exactly one of `while`/`until`; `loop.body` nodes are
      contiguous with no orchestrator node / hard gate between them (now enforced).
- [ ] Gate ids are unique and disjoint from node ids.
- [ ] Every `workflow:` target is in `imports`; validate with `--defs-root` to check child contracts.
- [ ] Human-interaction nodes are orchestrator, not background.
- [ ] No reliance on `customize` dotted-overrides or `grant_tools` (both inert).
- [ ] Run `validate-workflow` (expect `pass: true`) then `compile-workflow` and eyeball the
      `sequence` in the summary ŌĆö it should match the segments/checkpoints you intended.

```bash
.claude/scripts/validate-workflow  .claude/workflow-defs/<name>/WORKFLOW.yaml \
     .claude/workflow-defs/<name>/profiles/base.yaml
.claude/scripts/compile-workflow   <name>            # then read .compiled/manifest.json
```

`validate-workflow` exits `0` on pass, `1` on any block. `compile-workflow` exits `0` on success,
`1` on a schema/DAG error (with `schema_errors`), `2` on an environment error.
